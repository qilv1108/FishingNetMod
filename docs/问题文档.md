# 问题文档

生成日期：2026-06-09

## 范围

本文归档当前项目文档审查中发现的主要问题。重点范围为项目自身文档，不包含 `SMAPI/`、`.gstack-install/`、`.bun-ascii/`、`bin/obj` 等外部或生成目录。

## 总体判断

当前文档对 `0.1.0` 测试版发布已经基本够用，发布说明、手动测试结果、发布闸门和已知限制都有记录。主要问题不是缺少发布材料，而是历史文档、当前发布文档和测试报告之间缺少清晰的“当前权威来源”标注，后续维护者容易误读。

## 高优先级问题

### 1. 原始设计文档与当前实现不一致

根目录设计文档 `2026-05-20-fishing-net-implementation-design.md` 仍保留早期设计内容，例如铜网解锁写为钓鱼 2 级；当前手动测试清单和后续报告中则以钓鱼 1 级为准。

影响：后来维护者可能不知道应以早期设计还是当前实现为准。

建议：在该设计文档顶部补充状态说明，标记为“历史设计文档”；当前 `0.1.0` 行为以 `docs/release-notes/0.1.0.md`、`docs/reports/2026-06-08-0.1.0-manual-test-results.md` 和当前代码为准。

### 2. 完成报告保留了已解决问题

`docs/reports/2026-06-05-completion-report.md` 是历史快照，其中仍记录了一些后来已经解决的问题，例如：

- 根目录缺少 `.sln`。
- 自动测试基线为 70 个。
- 被动渔网仍使用原版纤维图标。
- 缺少真实游戏内验证记录。

当前仓库中 `FishingNet.sln` 已存在，发布文档记录测试基线为 74 个，手动测试结果也已经记录为 PASS。

影响：如果读者把该报告当成当前状态，会误判项目完成度。

建议：在报告顶部加“历史快照”说明，并指向最新手动测试报告和 release notes。

## 中优先级问题

### 3. 缺少项目根 README 入口

当前项目没有清晰的根级 `README.md` 作为入口。信息分散在根设计文档、release notes、manual test checklist、reports 和 `docs/superpowers/` 计划文档中。

影响：新接手者不知道应先看哪个文档，也不清楚哪些文档是历史资料、哪些是当前权威资料。

建议：新增根目录 `README.md`，至少包含：

- 项目简介。
- 当前版本：`0.1.0` 测试版。
- 构建/测试命令：`dotnet test`。
- 安装入口：`docs/release-notes/0.1.0.md`。
- 当前验证结果：`docs/reports/2026-06-08-0.1.0-manual-test-results.md`。
- 历史设计文档说明。

### 4. 手动测试清单与测试结果状态分离

`docs/manual-test-checklist.md` 中仍是未勾选模板状态，但 `docs/reports/2026-06-08-0.1.0-manual-test-results.md` 已记录 0.1.0 手动测试 PASS。

影响：读者只打开 checklist 时，可能误以为测试尚未执行。

建议二选一：

- 保持 checklist 作为模板，并在顶部注明实际 0.1.0 测试结果见报告文件；或
- 另存一份已勾选清单到 `docs/reports/`，作为实际测试归档。

### 5. 文档语言和文本策略不统一

当前文档中英文混用：release notes 和部分计划文档为英文，完成报告和手动测试清单为中文；游戏内 CP 文本主要为英文，C# HUD/错误提示有中文。

影响：不阻塞 `0.1.0` 测试版，但会影响长期维护和对外发布一致性。

建议：后续版本统一文档语言。若面向当前维护者，建议项目文档默认中文；游戏内文本再单独规划 i18n。

## 低优先级问题

### 6. 计划文档中未完成勾选项可能造成误导

`docs/superpowers/plans/` 下的部分计划文档仍保留大量 `- [ ]` 勾选项，但相关任务已经在后续提交或 release gate 中完成。

影响：这些文档作为执行计划保留没问题，但容易被误认为当前待办。

建议：在已完成或被替代的计划文档顶部标注状态，例如“已执行完成”“被 0.1.0 release gate 覆盖”或“历史计划”。

### 7. 历史设计文档中的结构名与当前代码不完全一致

早期设计文档中提到的部分类名和结构与当前代码不同，例如 `PassiveNetObject`、`NetPlacementManager` 等设计名，当前实现中已演化为 `PassiveNetManager`、`PassiveNetData`、`PassiveNetRenderer` 等。

影响：不影响发布，但会影响后续按文档查代码。

建议：后续整理一份当前架构说明，如 `docs/current-architecture.md`，或在根 README 中增加“当前代码结构”章节。

## 建议处理顺序

1. 给 `2026-05-20-fishing-net-implementation-design.md` 和 `docs/reports/2026-06-05-completion-report.md` 加历史状态说明。
2. 新增根 `README.md`，统一文档入口。
3. 明确 `docs/manual-test-checklist.md` 是模板还是实际结果。
4. 后续再整理当前架构文档和语言/i18n 策略。

## 结论

文档当前完成度约为 85%。`0.1.0` 测试版发布材料基本齐全，但需要补充“文档状态标注”和“入口 README”，以避免历史文档与当前实现互相冲突造成误读。
