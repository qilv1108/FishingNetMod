# 发布前四个关键任务设计

日期：2026-06-11

## 目标

完成当前发布前最关键的四项检查与说明，目标是让 `0.1.0` 测试版在发布前具备可追溯的验证结果和清晰的限制说明。

本设计覆盖四个任务：

1. 完整游戏内复测。
2. 发布包安装校验。
3. 多人模式限制说明。
4. 仓库临时文件确认。

## 范围

本轮先设计，再执行。执行阶段可以修改文档和新增验证报告，但不主动改游戏逻辑。

范围内：

- 运行自动化测试。
- 检查发布 zip 结构。
- 尝试本机启动游戏/SMAPI 并进入存档验证核心功能。
- 写入发布前验证记录。
- 更新 README 或 release notes 中的多人限制说明。
- 检查 `git status --short`，确认是否有未跟踪或临时文件。

范围外：

- 不实现多人模式完整规则。
- 不重做收网界面。
- 不新增主动撒网动画。
- 不实现固定 10% 被动垃圾概率。
- 不直接删除来源不明文件。

## 方案选择

采用严格发布闸门方案。

要求：

- 自动测试必须通过。
- 发布 zip 结构必须检查通过。
- 必须尝试启动游戏并进入存档。
- 至少验证 Mod/Content Patcher 加载无红色报错，并验证一个核心功能。
- 如果无法启动、无法进入存档或无法验证核心功能，本次不能标记为发布前通过，只能记录为 `BLOCKED`。

被否决方案：

- 半自动复核：由用户手动进游戏、我整理结果。该方案稳定，但本轮用户选择由我尝试本机验证。
- 文档复核：只基于已有 2026-06-08 PASS 记录。该方案最快，但不满足“必须进入存档并验证核心功能”的标准。

## 任务一：完整游戏内复测

执行阶段会尝试使用本机环境启动游戏或 SMAPI。

最低通过标准：

1. 成功启动到游戏并进入存档。
2. Mod 与 Content Patcher 加载无红色报错。
3. 至少验证一到两个核心功能，例如：
   - `fishing_net give copper` 能给到铜网；或
   - 主动撒网可打开数字挑战并发放捕获物；或
   - 被动渔网可以放置到水域。

产物：

- 新增发布前验证记录：`docs/reports/2026-06-11-pre-release-check.md`。

结果状态只允许使用明确状态：

- `PASS`：进入存档并完成核心功能验证。
- `BLOCKED`：无法启动、无法进入存档、无法交互或无法完成核心功能验证。
- `FAIL`：成功执行验证，但发现实际功能错误。

不得把“构建成功”或“已有旧报告通过”写成新的游戏内复测通过。

## 任务二：发布包安装校验

执行阶段运行：

```bash
dotnet test
```

该命令应完成构建、测试，并生成调试发布包：

```text
FishingNetMod/bin/Debug/net6.0/FishingNetMod 0.1.0.zip
```

zip 内必须包含：

- `FishingNetMod/FishingNetMod/manifest.json`
- `FishingNetMod/FishingNetMod/FishingNetMod.dll`
- `FishingNetMod/[CP] FishingNetMod/manifest.json`
- `FishingNetMod/[CP] FishingNetMod/content.json`
- `FishingNetMod/[CP] FishingNetMod/assets/fishing_net.png`
- `FishingNetMod/[CP] FishingNetMod/assets/i18n/default.json`

同时校验 release notes 中记录的安装结构与实际 zip 结构一致。

验证结果写入 `docs/reports/2026-06-11-pre-release-check.md`。

## 任务三：多人模式限制说明

本轮不实现多人模式改造，只把当前边界写清楚。

文档需要明确：

- 当前版本主要验证单人核心玩法。
- 多人模式仍是实验性。
- 以下多人规则未在 `0.1.0` 中完整定义或验证：
  - 其他玩家是否允许收取别人放置的网。
  - 被动渔网每日产出应计入哪个玩家的任务进度。
  - 多人存档中配方解锁进度是否应按玩家独立。

可能更新位置：

- `README.md`
- `docs/release-notes/0.1.0.md`
- `docs/reports/2026-06-11-pre-release-check.md`

文档措辞应避免暗示多人模式已经完整支持。

## 任务四：仓库临时文件确认

执行阶段检查：

```bash
git status --short
```

规则：

- 如果输出为空，记录为仓库干净。
- 如果存在未跟踪文件或修改文件，先列出文件和判断依据。
- 对明显由本轮生成的验证报告或文档更新，按正常改动处理。
- 对来源不明的图片、日志、临时文件，不直接删除；记录为需要用户确认。

本任务的完成标准不是“强行清空所有文件”，而是发布前没有未解释的临时文件。

## 错误处理

- 自动测试失败：发布包校验不能通过，报告记录失败输出摘要。
- zip 缺文件：发布包校验为 `FAIL`，列出缺失路径。
- 游戏无法启动：游戏内复测为 `BLOCKED`，记录命令、现象和下一步。
- 无法进入存档：游戏内复测为 `BLOCKED`，记录阻塞点。
- 能进入游戏但核心功能失败：游戏内复测为 `FAIL`，记录复现步骤。
- git 状态不干净：不隐瞒、不强删，记录文件列表和处理建议。
- 文档与代码冲突：以当前代码、测试和实际验证为准，文档标为待修正。

## 验收标准

本轮任务完成后，应满足以下条件：

1. `dotnet test` 已运行，并在报告中记录结果。
2. 发布 zip 结构已检查，并在报告中记录结果。
3. 游戏内复测结果明确为 `PASS`、`BLOCKED` 或 `FAIL`。
4. README 或 release notes 明确多人模式仍是实验性，并列出未定义规则。
5. `git status --short` 已检查，且报告说明是否存在临时文件。
6. 如果任何关键项不是 `PASS`，最终结论不得写成“发布前全部通过”。

## 实施顺序

1. 运行自动化测试并生成 zip。
2. 检查 zip 内容。
3. 尝试启动游戏/SMAPI 并进入存档验证核心功能。
4. 更新多人模式限制说明。
5. 写入发布前验证报告。
6. 检查仓库状态，确认没有未解释临时文件。
7. 根据结果给出最终发布前结论。
